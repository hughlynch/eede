eede: An Open Source Earth Engine Development Environment

A vision for modern EE tooling, built by and for the community.

GitHub repo: github.com/hughlynch/eede

The Opportunity

Google Earth Engine has ~500,000 registered developers and powers critical work in climate science, agriculture, forestry, disaster response, and urban planning. The Code Editor — the primary IDE for EE development — was groundbreaking when it launched over a decade ago. But it hasn't kept pace with modern development practices, and the community feels it.

Common frustrations from the EE community:

  • No version control integration. Scripts live in a proprietary file system with no git support.
  • No debugging. print() is the only debugging tool. No breakpoints, no variable inspection, no step-through.
  • JavaScript only in the IDE. The Python API is growing fast (especially for ML/AI workflows), but has no equivalent IDE experience.
  • No cells or notebooks. Every script is a monolithic block. No way to run sections independently.
  • Batch system limitations. Export tasks are fire-and-forget with poor monitoring, no dependency chains, and no retry logic.
  • No extensibility. Can't add custom panels, tools, or integrations. The UI is frozen in 2015.
  • Vendor lock-in concerns. Scripts, assets, and workflows are tightly coupled to Google infrastructure.

Meanwhile, the EE JavaScript API is fully open source (@google/earthengine on npm, google/earthengine-api on GitHub) and works perfectly as a standalone browser library. The API does all the heavy lifting — the Code Editor is "just" the IDE around it. This means building a better one is not only possible, it's practical.

What eede Is

eede (pronounced "Edie") is a modern, open source development environment for Earth Engine. It's not a clone of the Code Editor — it's what comes next.

Core Principles

  1. Dual-language from day one. JavaScript and Python side by side, in the same notebook. EE's serialization format means computations defined in either language produce identical server-side execution plans.
  2. Cell-based notebooks. Like Jupyter, but purpose-built for geospatial. Each cell can be JS or Python. Map layers persist across cells.
  3. Real map rendering. A full Leaflet/Deck.gl map with EE tile layers, drawing tools, inspector, and layer management.
  4. Modern editing. Monaco editor (the engine behind VS Code) with EE-aware autocomplete and inline documentation.
  5. Open and extensible. Plugin architecture for custom panels, tools, data sources, and export targets.

Key Technical Decisions

  • EE API as a browser library. The npm package works standalone. No server-side proxy needed. Authentication via OAuth or service account tokens.
  • Cell execution model. Each cell runs in a scoped context with shared state. JS cells execute via the browser EE API. Python cells via Pyodide (WebAssembly) or a local kernel.
  • Cross-language bridge. EE's computation model is language-agnostic. Variables can be bridged between JS and Python by serializing/deserializing the EE computation graph.
  • Map rendering. Leaflet for 2D, Deck.gl for 3D/WebGL. EE tile layers via getMapId(). Drawing tools and click-to-inspect.
  • Storage. Notebooks as files in the local filesystem or git. No proprietary storage.

The Dual-Language Story

This is eede's most important feature. ~80% of existing EE code is JavaScript. The Python API is growing, especially for ML/AI, but most EE knowledge, tutorials, and shared scripts are in JS. Forcing users to choose one language is a non-starter.

Why it works:

  1. EE computations are language-agnostic. Both JS and Python APIs serialize to the same intermediate representation.
  2. EE objects can be serialized. ee.Serializer converts computation graphs to JSON; ee.Deserializer reconstructs them. Variables cross the language boundary losslessly.
  3. The map is shared. Map.addLayer() in either language adds layers to the same Leaflet map.

Migration path:

  • Week 1: Copy Code Editor scripts directly into eede. They run as-is in JS cells.
  • Week 2: Add Python cells for new analysis steps. Reference JS-defined variables from Python.
  • Month 2: Gradually port utilities to Python where it makes sense. Use Python for ML, pandas, matplotlib.
  • Ongoing: Use whichever language is best for each task.

Batch System

The Code Editor's Export system is one of its weakest points. eede's batch system would provide:

  • Task graphs. Define dependencies between exports.
  • Retry with backoff. Automatic retry on transient failures.
  • Progress monitoring. Real-time progress, estimated completion, cost tracking.
  • Scheduling. Run exports on a schedule.
  • Local pre-processing. Local Python/JS steps as part of the pipeline.
  • Templates. Reusable export configurations.

Extensibility Framework

A plugin API for: custom panels, data source plugins (S3, STAC catalogs), export targets (PostGIS, tile servers), analysis tools (fire detection, crop classification), and themes/layouts.

What We've Already Proven

  1. EE JS API works standalone in a browser. Loaded from npm, initialized with a gcloud token, runs all standard operations.
  2. Real tile rendering works. getMapId() returns tile URLs that render correctly on Leaflet. Verified with 100+ EE computations.
  3. The full Code Editor API surface can be mocked. Map.addLayer, Export.image.toDrive, ui.Chart, print() — all implementable outside the Code Editor.
  4. 99/103 golden test fixtures pass in our browser harness.
  5. Authentication is straightforward. gcloud tokens and service account keys both work.

The Strategic Case

For Google: An active open source EE IDE community means more EE adoption, more compute on Google Cloud, and less maintenance burden. Google doesn't monetize the Code Editor — they monetize the compute. A better IDE drives more compute.

For the community: Modern tooling, version control, dual-language support, debugging, extensibility. The ability to build the features they've been requesting for years.

For science: Reproducible workflows (notebooks in git), better collaboration (PRs instead of shared scripts), and lower barriers to entry (Python support for the ML/AI crowd).
